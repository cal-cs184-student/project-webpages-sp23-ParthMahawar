Part 1

De Casteljau’s algorithm finds points at a given parameter along a Bezier curve by recursively interpolating between given control points. It follows the below steps:

  • Given parameter t and n control points, create n-1 control points by calculating a weighted average of adjacent points, weighting the left point by (1-t) and the right point by t.
  • Repeat the above step until only 1 point is left.
  • That one point is the point at parameter t on the entire bezier curve.

Part 2

The algorithm extends to surfaces defined by a grid of points by essentially treating each row of points as a Bezier curve, to get a new column bezier curve with the point on each row curve as control points for the column curve.

This was implemented in pretty much that exact way given parameters (u, v) - using De Casteljau’s algorithm to first calculate the point on the Bezier curve defined by each row’s control points at u, then using the points for each row as control points for another bezier curve, evaluating it at parameter v, and returning that value.

Part 3

To implement area-weighted vertex normals, I first obtained all the neighbors of the vertex by traversing half-edges. Then, as adjacent members of this list would share a face with the original vertex, I took the cross product of the two vectors defined by the original vertex and each of two adjacent neghbors, which I then divided by its magnitude to get a normalized normal. I also used the 3 points to calculate the area of the triangle. I repeated this project for all adjacent neighbors of the vertex to obtain an array of normals and an array of their corresponding areas. I then calculated an average of the normals weighted by those areas to obtain the area-weighted vertex normal.

Part 4

I had a rather standard implementation for edge flip, which simply involved reassigning all halfedge, edge, vertex, and face pointers in such a way that the mesh was now defined as if the edge was flipped. I did have problems initially of holes being created in the mesh, but this was fixed when instead of trying to pick and choose which pointers to reassign I simply reassigned all the ones that had any chance of changing - so chances are I had just missed something in my initial implementation.

Part 6

I implemented loop subdivision by following the recommendation in the spec - calculating new positions for existing vertices first, then calculating new positions for new vertices, and finally splitting then flipping. The major implementation trick that I had to use to get things working was adding a variable isBeingSubdivided to the edge class, so that I could reuse the same split function and have it decide whether to add a vertex to the midpoint, or to the edge→newPosition calculated.

The major thing I’ve noticed is that vertices tend to get a bit closer together after subdivision, and the mesh typically decreases in volume. Sharp corners and edges tend to round off, which makes sense as the entire point of subdivision is to make these transitions smoother.

Cube becoming asymmetric after several subdivisions - this behavior occurs as the positions of existing vertices are updated as a weighted average of their current positions and those of their neighbors. The vertices that have diagonal edges connected to them are therefore pulled further into the cube as they have more neighbors that are further away.

By splitting the edge on every face of the cube, the cube no longer subdivides asymmetrically, as now all vertices have an equal number of neighbors, at the same distance across the cube.
